In the instance of a PDF e-form (or similar) where the meta data is held within the form itself, an automated analysis function should be invoked to infer any required meta data. (Note that this can be overridden at a later point).<br /></p><p><br /></p><p>The meta data should be provided with a unique identification number, and persisted. (The persistence method is up to the implementor, but could involve the use of a database or file store).<br /></p><p><br /></p><p>The form and overlay are initially related, however can be completely decoupled. <br /></p>

Scenario Type
Basic Path Basic Path
Details:
Steps:
1. Middleware analysis document for meta data, including document ID, type, overlay method and form meta data should it exist
2. Middleware validates that the document is of a type that can be used
3. Middleware creates the document in a Form Register
4. Check if document includes form metadata
5. Results are written to a notification file and dropped in the document interchange
Exceptions:
1a. Document metadata is unavailable (at End)
2a. Document is invalid (at End)
3a. An error occurs writing information to the database (at End)
4a. DocumentIncludesMetadata (at 5)
DocumentIncludesMetadata Alternate
Details:
Steps:
1. Middleware registers metadata in database
Document metadata is unavailable Exception
Details:
Steps:
1. Middleware collates error information
2. Middleware writes results to a log file.
Document is invalid Alternate
Details:
Steps:
1. Middleware collates error information
2. Middleware writes results to a log file.

Details:

1. Middleware collates error information
2. Middleware records error information in a log file
