Standards provide a common set of rules that should be applied to the various elements within the CCA.  To achieve rigor it is necessary to apply them consistently by review of all components before they are implemented.  Too many standards can be just as disabling as having no standards at all.  <br /></p><p><br /></p><p>It is important for STL to define the standards that they would like to adhere to.  This does not need to be perfect from the outset, but it is important to test any standard selection and try to "fail fast”.<br /></p><p><br /></p><p>At the time of writing, Data governance standards are being defined.<br /></p><p><br /></p><p>The governance team should have a clear understanding of the following domains of standards:<br /></p><p><br /></p><p><ul><li>Naming conventions for files, database and XML schemas, entities, attributes, tables, fields, message formats, etc.</li><br /></p><p><li>Messaging protocols and SOA-specific standards including business- and technical-level service definitions</li><br /></p><p><li>Standards for developing data models, process models, class models, etc. (UML, EMF, and others)</li><br /></p><p><li>Data access standards (SQL, JDBC, ODBC, etc.)</li><br /></p><p><li>Transactional integrity standards, e.g., XA protocol for distributed transaction processing</li><br /></p><p></ul><br /></p>
